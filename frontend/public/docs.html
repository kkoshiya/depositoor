Getting Started

depositoor is an EIP-7702 deposit sweeper that accepts any token on any supported chain and settles the equivalent value in USDC at your destination address. Drop the widget into your app with three lines of code.

Installation

Terminal
npm install @depositoor/react

Quick Start

App.tsx
import { DepositoorProvider, DepositWidget } from '@depositoor/react'

export default function App() {
  return (
    <DepositoorProvider
      apiUrl="https://api.depositoor.xyz"
      implementationAddress="0x33333393A5EdE0c5E257b836034b8ab48078f53c"
    >
      <DepositWidget
        destinationAddress="0x123...abc"
        destinationChainId={8453}
      />
    </DepositoorProvider>
  )
}

That is the complete integration. The widget handles wallet generation, EIP-7702 authorization signing, session registration, deposit detection, and real-time status updates via SSE. The user sends any token on any supported chain, and USDC arrives at the destination.

How It Works (30 seconds)

  1. 1. The SDK generates a fresh burner wallet and signs an EIP-7702 authorization delegating code execution to the DepositoorDelegate contract (chainId=0 means valid on all chains).
  2. 2. A session is registered with the backend, which begins indexing all supported chains for incoming transfers to that burner address.
  3. 3. The user sends any token (or native ETH) to the burner address on any supported chain. The indexer detects the deposit in real time via WebSocket block subscriptions.
  4. 4. The sweeper uses the signed EIP-7702 authorization to execute batched calls from the burner: wrap ETH if needed, swap to USDC via Uniswap, and bridge cross-chain via Across -- all in 1-2 transactions.
  5. 5. USDC settles at the destination address on the destination chain. The SDK receives the final status via SSE and fires the onComplete callback.

Architecture

The system consists of four components: the React SDK (client), the Rust backend (API + indexer + sweeper), the Uniswap Trading API (swap/bridge execution), and the DepositoorDelegate smart contract (ERC-7821 batch executor deployed on all chains).

Process Flow

user sends token ---> indexer detects ---> sweeper executes ---> USDC at destination (any chain) (WS + HTTP) (EIP-7702 batch) (any chain)

Session Lifecycle

Each deposit session transitions through a set of statuses tracked in PostgreSQL and pushed to the client in real time via SSE:

pending → detected → sweeping → bridging → swept

If anything fails, the session moves to failed with an error message. Sessions that receive no deposit within 30 minutes are considered expired. Failed sessions can be refunded via the POST /sessions/:id/refund endpoint.

DepositoorProvider

The context provider that must wrap any depositoor components. It supplies the API URL and the implementation (delegate) contract address to all child components via React context.

Usage
import { DepositoorProvider } from '@depositoor/react'

<DepositoorProvider
  apiUrl="https://api.depositoor.xyz"
  implementationAddress="0x33333393A5EdE0c5E257b836034b8ab48078f53c"
>
  {children}
</DepositoorProvider>

Props

Prop Type Required Description
apiUrl string Yes Base URL of the depositoor backend API
implementationAddress `0x${string}` Yes Address of the DepositoorDelegate contract. Use 0x33333393A5EdE0c5E257b836034b8ab48078f53c for production on all chains.
children ReactNode Yes Child components that consume the context

DepositWidget

The drop-in deposit UI component. Handles the full flow: burner key generation, EIP-7702 authorization, session creation, QR code display, deposit detection via SSE, and status updates. Must be rendered inside a DepositoorProvider.

Full Example

function PaymentPage() {
  return (

      apiUrl="https://api.depositoor.xyz"

    >

Session Status Values

Status Description

useDepositoor Hook

For advanced integrations where you need direct access to the depositoor context without using the pre-built widget. Returns the API URL and implementation address from the nearest DepositoorProvider.

useDepositoor.ts
import { useDepositoor } from '@depositoor/react'

function CustomDeposit() {
  const { apiUrl, implementationAddress } = useDepositoor()

  // Use apiUrl to call session endpoints directly
  // Use implementationAddress for EIP-7702 authorization

  const createSession = async () => {
    const res = await fetch(`${apiUrl}/sessions`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        burner_address: burnerAddress,
        eip7702_auth: signedAuth,
        destination_address: destAddress,
        destination_chain: 8453,
      }),
    })
    return res.json()
  }

  // ...
}

Return Value

Field Type Description
apiUrl string Backend API base URL from the provider
implementationAddress `0x${string}` DepositoorDelegate contract address for EIP-7702 auth signing

EIP-7702 Authorization

When building a custom integration, you need to generate the EIP-7702 authorization yourself. The authorization delegates the burner EOA's code to the DepositoorDelegate contract, allowing the relayer to call execute() on the burner as if it were the contract.

EIP-7702 Auth Signing (viem)
import { generatePrivateKey, privateKeyToAccount } from 'viem/accounts'

const privateKey = generatePrivateKey()
const burner = privateKeyToAccount(privateKey)

// chainId: 0 means valid on ALL chains
const auth = await burner.signAuthorization({
  contractAddress: implementationAddress,
  chainId: 0,  // wildcard -- works on any chain
  nonce: 0,
})

// auth = { address, chainId, nonce, r, s, yParity }

Setting chainId: 0 is critical. It means the user's single signature authorizes sweep execution on any supported chain, so regardless of which chain they deposit on, the relayer can act.

GET /sessions/:id/events

Server-Sent Events (SSE) stream for real-time session updates. Sends the current state immediately on connect, then pushes updates on every status change via PostgreSQL LISTEN/NOTIFY. Keep-alive pings every 15 seconds.

Event Format

event: status
data: {"id":"a1b2c3d4-...","status":"detected","detected_token":"0xaf88d065...","detected_amount":"1000000000",...}

event: status
data: {"id":"a1b2c3d4-...","status":"sweeping",...}

event: status
data: {"id":"a1b2c3d4-...","status":"swept","sweep_tx":"0xdef456...",...}

Client Usage

const source = new EventSource(`${apiUrl}/sessions/${sessionId}/events`)

source.addEventListener('status', (e) => {
  const session = JSON.parse(e.data)
  console.log(session.status) // "pending" | "detected" | "sweeping" | ...
})

Indexer

The indexer monitors all supported chains for incoming deposits to burner addresses. It uses a hybrid WebSocket + HTTP polling approach for reliability.

Hybrid WS + HTTP Design

For each chain, the indexer runs a watcher that follows the chain head:

  1. 1. Primary: WebSocket newHeads subscription. Subscribes to new block headers via a WebSocket RPC connection. For each new block, it fetches Transfer(address,address,uint256) event logs via HTTP RPC.
  2. 2. Fallback: HTTP polling. If the WS connection drops, the indexer falls back to polling eth_blockNumber every 2 seconds and processing the latest block via HTTP. It automatically reconnects WS in the background.
  3. 3. Native ETH detection. In addition to ERC-20 Transfer log scanning, the indexer polls eth_getBalance for all pending burner addresses each block. If a burner has a non-zero ETH balance, it is claimed with the WETH address as the detected token (the sweeper will wrap it).

Detection Flow

new block ---> get_logs(Transfer) ---> match topic[2] ---> claim_for_detection() against pending session burners new block ---> eth_getBalance(burners) ---> if > 0 ---> claim_for_detection(WETH)

The claim_for_detection database call uses an atomic update to prevent duplicate processing. Only one indexer instance can claim a session. The session status transitions from pending to detected, recording the source chain ID, token address, amount, and deposit transaction hash.

Sweeper

The sweeper executes deposit sessions once the indexer detects an incoming transfer. It determines the optimal execution path based on the source token and whether the deposit is same-chain or cross-chain.

The 4 Sweep Flows

Source Token Destination Transactions Execution
USDC Same chain 1 tx sweep(USDC, dest) -- direct transfer
Non-USDC ERC-20 Same chain 1 tx approve + Uniswap swap (set swapper=dest so output goes directly to destination)
USDC Cross-chain 1 tx approve + Uniswap BRIDGE route (Across Protocol)
Non-USDC ERC-20 Cross-chain 2 tx TX1: swap to USDC (output stays at burner). TX2: bridge USDC to dest chain.
Native ETH Any +1 call Prepend WETH.deposit{value: amount}() then follow the WETH flow above

EIP-7702 Batched Execution

All calls within a single transaction are batched using the ERC-7821 execute() standard. The relayer sends a Type-4 (EIP-7702) transaction with the stored signed authorization and the batched calldata:

Batch Structure
// mode = 0x01 (batch mode)
// executionData = ABI-encoded array of (address, value, calldata) tuples

execute(
  0x0100...00,   // mode: batch
  abi.encode([
    (WETH,   amount,  WETH.deposit()),       // wrap ETH (if native)
    (token,  0,       approve(router, amt)),  // approve Uniswap router
    (router, 0,       swap(...)),              // execute swap
  ])
)

For the cross-chain non-USDC case, two transactions are needed because the bridge call requires knowing the exact USDC amount, which is only known after the swap confirms. The second transaction does not need to re-send the EIP-7702 authorization because the delegation persists on the burner's account until the nonce increments.

The Swapper Trick

For same-chain swaps, the sweeper sets the swapper parameter in the Uniswap quote to the destination address. This makes Uniswap route the swap output directly to the final recipient in a single transaction, avoiding an extra transfer step. For cross-chain flows, the first swap sets swapper=burner so USDC stays at the burner for the subsequent bridge call.

Uniswap Integration

depositoor uses the Uniswap Trading API (v1) for all swap and bridge execution. The integration uses the proxy approval flow (no Permit2) to work within the EIP-7702 delegation model.

Proxy Approval Flow

Because the burner is an EOA with delegated code (not a contract wallet with Permit2 support), we disable Permit2 and use direct token approvals instead:

Header Value Purpose
x-permit2-disabled true Disables Permit2 signatures. The API returns a proxy contract address for direct approve + transferFrom flow.
x-universal-router-version 2.0 Use Universal Router v2 for execution.
x-chained-actions-enabled true Enable cross-chain routing (set only for bridge quotes).

API Flow

  1. 1. Quote: POST /v1/quote with token pair, amount, and slippage_tolerance: 0.5. For bridges, tokenInChainId != tokenOutChainId.
  2. 2. Strip Permit2 data: Remove permitData and permitTransaction from the quote response before sending to /swap.
  3. 3. Swap calldata: POST /v1/swap returns the final transaction (to, data, value). The to address is the router/proxy contract.
  4. 4. Execute: The sweeper batches approve(router, amount) + the swap calldata into a single ERC-7821 execute call.

BRIDGE Routing

For cross-chain USDC transfers, the Uniswap API returns a "routing": "BRIDGE" response that uses Across Protocol under the hood. The sweeper validates that the routing type is BRIDGE and rejects any non-bridge responses. Bridge fills typically complete in 2-10 seconds on L2s.

Bridge Quote Example
// Quote for USDC on Arbitrum -> USDC on Base
const quote = await fetch('https://trade-api.gateway.uniswap.org/v1/quote', {
  method: 'POST',
  headers: {
    'x-api-key': API_KEY,
    'x-permit2-disabled': 'true',
    'x-chained-actions-enabled': 'true',
  },
  body: JSON.stringify({
    swapper: destinationAddress,  // output goes here on dest chain
    tokenIn: '0xaf88d065...',    // USDC on Arbitrum
    tokenOut: '0x833589fC...',   // USDC on Base
    tokenInChainId: '42161',
    tokenOutChainId: '8453',
    amount: '1000000000',
    type: 'EXACT_INPUT',
  })
})
// response.routing === "BRIDGE"

WalletConnect / Reown

The depositoor widget includes a WalletConnect integration via Reown's AppKit. Users can connect their existing wallet (MetaMask, Rainbow, Coinbase Wallet, etc.) and send tokens directly to the burner address from the connected wallet.

How It Works

  1. 1. User clicks "Connect Wallet" in the widget, which opens the Reown AppKit modal with a QR code (for mobile wallets) or direct connection (for browser extensions).
  2. 2. Once connected, the widget can request a sendTransaction to transfer the selected token to the burner address on the chosen source chain.
  3. 3. The indexer detects the incoming transfer, and the rest of the flow continues as usual: sweep, bridge, settle.

Multi-Chain Support

The AppKit integration supports all 5 chains. The user selects which chain to send from in the widget UI, and the WalletConnect session is configured for that chain. If the user's wallet is on a different chain, it will prompt a chain switch before the transfer.

QR Code Flow (Mobile)

For mobile users, the widget displays a QR code that opens their wallet app. The QR code is generated using the cuer library for the burner address itself (for simple transfers) or using the WalletConnect pairing URI (for wallet-assisted transfers). This allows users to scan and send from any mobile wallet without installing browser extensions.

DepositoorDelegate

The core smart contract. An ERC-7821 minimal batch executor that gets delegated to via EIP-7702. When a user signs an EIP-7702 authorization, their EOA temporarily "becomes" this contract, allowing the relayer to execute arbitrary batched calls from the user's address.

DepositoorDelegate.sol
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

import {ERC7821} from "solady/accounts/ERC7821.sol";

interface IERC20 {
    function balanceOf(address) external view returns (uint256);
    function transfer(address, uint256) external returns (bool);
    function approve(address, uint256) external returns (bool);
}

contract DepositoorDelegate is ERC7821 {
    address public immutable weth;
    address public immutable keeper;

    constructor(address _weth, address _keeper) {
        weth = _weth;
        keeper = _keeper;
    }

    /// @notice Sweep full balance of a token to a recipient.
    function sweep(address token, address to) external {
        require(msg.sender == keeper || msg.sender == address(this), "unauthorized");
        uint256 bal = IERC20(token).balanceOf(address(this));
        if (bal > 0) {
            require(IERC20(token).transfer(to, bal), "transfer failed");
        }
    }

    function _execute(
        bytes32,
        bytes calldata,
        Call[] calldata calls,
        bytes calldata opData
    ) internal virtual override {
        if (opData.length == 0) {
            require(
                msg.sender == keeper || msg.sender == address(this),
                "unauthorized"
            );
            _execute(calls, bytes32(0));
            return;
        }
        revert("opData not supported");
    }

    receive() external payable override {
        (bool success,) = weth.call{value: msg.value}("");
        require(success, "WETH wrap failed");
    }
}

Function Reference

Function Access Description
sweep(token, to) keeper | self Transfer the entire balance of token held by this address to to. Used for same-chain USDC deposits and refunds.
_execute(calls, opData) keeper | self ERC-7821 batch execution override. Accepts an array of (address, value, calldata) tuples and executes them sequentially. The opData parameter must be empty (reserved for future use).
receive() anyone Auto-wraps incoming ETH to WETH by calling WETH.deposit{value}(). This ensures native ETH deposits are immediately convertible by the sweeper.

Immutable State

Variable Description
weth WETH contract address on this chain. Used by receive() to auto-wrap ETH. On OP Stack L2s (Base, Optimism, Arbitrum): 0x4200000000000000000000000000000000000006.
keeper The relayer address authorized to call sweep() and execute(). Only this address (or the contract itself via batch self-calls) can move funds.

Security Model

The contract is deliberately minimal (48 lines). Access control restricts all fund-moving operations to the keeper address (the relayer) or address(this) (self-calls within a batch). The opData path is explicitly reverted to prevent unauthorized execution modes. The contract inherits from Solady's ERC7821 which provides the gas-optimized batch call execution. Deployed via CREATE3 to the same address on all chains.

Deployments

DepositoorDelegate is deployed at the same address on all chains via CREATE3 (salt-deterministic deployment). This allows the EIP-7702 authorization to use chainId: 0 with a single implementation address.
